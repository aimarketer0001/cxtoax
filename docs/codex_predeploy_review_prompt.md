# Codex 배포 전 최종 검토 요청 프롬프트

아래 프롬프트는 **Codex에서 프로젝트를 배포하기 전에 최종 검토를 요청할 때** 사용할 수 있는 프롬프트입니다.

권장 사용 상황:

- Next.js / React 기반 웹 프로젝트
- TypeScript 사용 프로젝트
- Supabase / PostgreSQL 연동 프로젝트
- Vercel 배포 예정 프로젝트
- 배포 직전 기능, UI, 보안, 환경 변수, 빌드 오류를 한 번에 점검하고 싶은 경우

---

```md
# 배포 전 최종 검토 및 수정 요청

당신은 시니어 풀스택 개발자이자 배포 전 품질 검토 담당자입니다.

현재 프로젝트는 운영 환경 배포 직전 상태입니다.  
단순 코드 리뷰가 아니라, 실제 사용자가 접속했을 때 문제가 발생하지 않도록 **기능, UI/UX, 문구, 코드 품질, 보안, 성능, Supabase/PostgreSQL 연동, Vercel 배포 안정성** 관점에서 전체 프로젝트를 점검해 주세요.

가능한 문제는 직접 수정하고, 수정이 어렵거나 확인이 필요한 항목은 별도로 정리해 주세요.

## 1. 프로젝트 전제

이 프로젝트는 다음 기술 스택을 기준으로 검토해 주세요.

- Frontend: Next.js, React, TypeScript
- Styling: Tailwind CSS 또는 CSS Modules
- Backend/API: Next.js API Routes 또는 Server Actions
- Database: Supabase / PostgreSQL
- Auth: Supabase Auth 사용 가능성 있음
- Storage: Supabase Storage 사용 가능성 있음
- Deployment: Vercel
- Package Manager: package.json을 확인하여 npm, pnpm, yarn 중 실제 사용 중인 것을 기준으로 판단

프로젝트 구조와 실제 파일을 먼저 확인한 후, 위 전제가 다를 경우 실제 코드 기준으로 검토해 주세요.

## 2. 검토 목표

다음 질문에 답할 수 있을 정도로 프로젝트를 점검해 주세요.

1. 지금 상태로 배포해도 되는가?
2. 배포하면 사용자가 핵심 기능을 정상적으로 이용할 수 있는가?
3. 빌드, 린트, 타입 체크에서 문제가 없는가?
4. Supabase / PostgreSQL 연결이 운영 환경에서도 안전하게 작동하는가?
5. Vercel 환경 변수와 배포 설정에 문제가 없는가?
6. 보안 정보, 테스트 데이터, 임시 코드가 노출되어 있지 않은가?
7. 모바일과 데스크톱에서 화면이 깨지지 않는가?
8. 사용자가 이해하기 어려운 문구나 미완성 문구가 남아 있지 않은가?

## 4. 실행 및 빌드 검토

가능하다면 아래 명령을 실제 프로젝트에 맞게 실행해 주세요.

package.json을 확인한 뒤 존재하는 스크립트만 실행해 주세요.

예시:

```bash

npm run lint

npm run build
npm run test
```

pnpm 또는 yarn을 사용하는 경우 프로젝트에 맞게 변경해 주세요.

확인할 항목:

- 의존성 설치 오류 여부
- lint 오류 여부
- TypeScript 타입 오류 여부
- Next.js build 오류 여부
- 테스트 실패 여부

- browser-only API 사용 위치 오류 가능성
- window, document, localStorage 사용 위치 오류 가능성
- dynamic import가 필요한 부분 여부

## 5. Next.js / React 검토 항목

다음 항목을 중점적으로 확인해 주세요.

### 라우팅

- app router 또는 pages router 구조가 일관적인지
- 존재하지 않는 페이지로 이동하는 링크가 없는지
- 동적 라우트 파라미터 처리에 문제가 없는지
- not-found.tsx, error.tsx, loading.tsx가 필요한 위치에 있는지
- 페이지 이동 시 사용자 흐름이 끊기지 않는지

### 컴포넌트

- 컴포넌트 역할이 명확한지
- 지나치게 큰 컴포넌트는 없는지
- 재사용 가능한 UI가 중복 구현되어 있지 않은지
- props 타입이 명확한지
- any 타입이 과도하게 사용되지 않았는지
- 상태 관리가 불필요하게 복잡하지 않은지

### Client / Server Component

- 클라이언트 컴포넌트에만 필요한 곳에 `"use client"`가 선언되어 있는지
- 서버 컴포넌트에서 브라우저 전용 API를 사용하지 않는지
- 클라이언트에 노출되면 안 되는 서버 로직이 client component로 들어가 있지 않은지
- 서버 액션이나 API Route에서 예외 처리가 적절한지

### 데이터 페칭

- 중복 API 호출이 없는지
- 로딩 상태가 처리되어 있는지
- 에러 상태가 처리되어 있는지
- 빈 데이터 상태가 처리되어 있는지
- 캐싱 전략이 적절한지
- revalidate, no-store, dynamic 설정이 의도에 맞는지

## 6. Supabase / PostgreSQL 검토 항목

Supabase를 사용하는 경우 다음 항목을 반드시 확인해 주세요.

### 환경 변수

- `NEXT_PUBLIC_SUPABASE_URL` 사용이 적절한지
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` 사용이 적절한지
- `SUPABASE_SERVICE_ROLE_KEY`가 클라이언트 코드에 노출되어 있지 않은지
- service role key는 서버 전용 코드에서만 사용되는지
- .env 파일이 Git에 포함되지 않았는지
- .env.example에는 실제 비밀값이 들어 있지 않은지

### Supabase Client

- browser client와 server client가 구분되어 있는지
- 클라이언트 컴포넌트에서 server-only key를 사용하지 않는지
- 인증 세션 처리가 올바른지
- 쿠키 기반 인증을 사용하는 경우 server component와 middleware 처리가 맞는지

### Auth

- 로그인, 로그아웃, 회원가입, 세션 유지가 정상적인지
- 인증이 필요한 페이지에 접근 제어가 있는지
- 비로그인 사용자가 보호된 데이터에 접근할 수 없는지
- 로그인 실패, 세션 만료, 권한 부족 상황 안내가 있는지

### Database

- 테이블명과 컬럼명이 코드와 일치하는지
- insert, select, update, delete 쿼리에 에러 처리가 있는지
- nullable 컬럼 처리가 안전한지
- 날짜, 숫자, boolean 타입 처리 오류가 없는지
- pagination이 필요한 데이터에 무제한 조회가 없는지
- 중복 데이터 생성 가능성이 없는지

### RLS 정책

Supabase RLS를 사용하는 경우 다음을 확인해 주세요.

- 운영 테이블에 RLS가 활성화되어 있는지
- 사용자가 본인 데이터만 조회/수정/삭제할 수 있는지
- 공개 데이터와 비공개 데이터가 명확히 분리되어 있는지
- anon 사용자에게 과도한 권한이 열려 있지 않은지
- insert/update/delete 정책이 의도와 맞는지
- 관리자 권한 처리가 안전한지

### Storage

Supabase Storage를 사용하는 경우 다음을 확인해 주세요.

- 업로드 가능한 파일 타입 제한이 있는지
- 파일 크기 제한이 있는지
- 공개 bucket과 비공개 bucket이 적절히 구분되어 있는지
- 업로드 파일명이 충돌하지 않도록 처리되어 있는지
- 사용자가 다른 사용자의 파일에 접근할 수 없는지

## 7. Vercel 배포 검토 항목

다음 항목을 확인해 주세요.

### 배포 설정

- Vercel에서 사용할 build command가 package.json과 일치하는지
- output directory 설정이 필요한 경우 올바른지
- Node.js version 문제가 없는지
- runtime 설정이 필요한 API Route가 있는지
- Edge Runtime 사용 시 Node 전용 API를 사용하지 않는지

### 환경 변수

Vercel에 등록해야 할 환경 변수를 정리해 주세요.

다음 형식으로 정리해 주세요.

| 환경 변수명 | 사용 위치 | 필수 여부 | 클라이언트 노출 가능 여부 | 비고 |
|---|---|---:|---:|---|

확인할 항목:

- 로컬 .env와 Vercel 환경 변수가 일치하는지
- Production / Preview / Development 환경 변수 구분이 필요한지
- NEXT_PUBLIC 접두사가 필요한 변수와 필요 없는 변수가 구분되어 있는지
- 비밀값이 프론트엔드 번들에 포함될 가능성이 없는지

### 배포 후 확인 URL

배포 후 직접 확인해야 할 주요 URL 목록을 만들어 주세요.

예시:

| 구분 | URL 또는 경로 | 확인 항목 |
|---|---|---|

| 로그인 | /login | 로그인 가능 여부 |
| 대시보드 | /dashboard | 인증 후 접근 여부 |
| API | /api/... | 정상 응답 여부 |

## 8. UI/UX 검토 항목

다음 항목을 확인해 주세요.

### 화면 구성

- 첫 화면에서 서비스 목적이 명확한지
- 주요 CTA 버튼이 잘 보이는지
- 사용자가 다음 행동을 쉽게 알 수 있는지
- 페이지별 제목과 설명이 자연스러운지
- 화면 간 디자인 톤이 일관적인지

### 반응형

다음 해상도를 기준으로 레이아웃 문제를 확인해 주세요.

- 모바일: 360px ~ 430px
- 태블릿: 768px
- 데스크톱: 1280px 이상

확인할 항목:

- 가로 스크롤 발생 여부
- 카드, 표, 버튼, 입력폼 깨짐 여부
- 모바일 메뉴 동작 여부
- 긴 텍스트 줄바꿈 처리
- 모달 또는 팝업이 화면 밖으로 벗어나지 않는지

### 접근성

- 버튼에 접근 가능한 이름이 있는지
- input에 label 또는 aria-label이 있는지
- 이미지에 alt가 있는지
- 키보드 탐색이 가능한지
- 색상 대비가 지나치게 낮지 않은지
- focus 표시가 사라지지 않았는지

## 10. 보안 검토 항목

다음 항목을 반드시 확인해 주세요.

- API key, secret, token, password가 코드에 직접 노출되어 있지 않은지
- .env 파일이 Git 추적 대상이 아닌지
- 클라이언트 코드에 서버 전용 비밀값이 들어 있지 않은지
- 사용자 입력값을 그대로 HTML로 렌더링하지 않는지
- dangerouslySetInnerHTML 사용 시 sanitizing 처리가 있는지
- SQL injection 위험이 없는지
- 인증이 필요한 API에 권한 검사가 있는지
- 사용자가 다른 사람의 데이터에 접근할 수 없는지
- 관리자 기능이 일반 사용자에게 노출되지 않는지
- CORS 설정이 과도하게 열려 있지 않은지
- 업로드 기능이 있다면 파일 타입과 크기 제한이 있는지
